# Reason 551903 — Patch notes automation scope in THE-LARGER-SYSTEM

**Linked change:** [THE-LARGER-SYSTEM.md](../THE-LARGER-SYSTEM.md)

## Why `THE-LARGER-SYSTEM.md` was updated

Readers (including future agents) could infer from **“Release line: patch notes + semver on merge”** that **`npm run prepare-merge`** also **finishes** public patch notes for shipping—e.g. stripping draft placeholders or generating `public/*.md` from `reasons.md`. That is **not** what the script does today, which caused confusion when frozen versions still showed template lines in `/patchnotes`.

## What changed

The **Release line** subsection now states explicitly:

- **Hands-off / automated at merge today** = semver alignment, `UNRELEASED` → `<semver>` rename, rewriting **structural** `UNRELEASED` markers in frozen markdown, and seeding the next `UNRELEASED/` tree.
- **Still human-led** = audience copy in `public/*.md`, removal of draft/template lines, and any “empty tier” honest messaging—**before** merge (or by editing the frozen folder before a publishable client build).

**Public patch notes** now cross-links to that scope so voice/outline guidance is not read as implying full prose automation.

## How this serves the larger system

- **Coherence:** One document distinguishes **release bookkeeping** (script) from **editorial truth** (humans), without contradicting [patchnote/README.md](../../patchnote/README.md) or [patchnotes-release.md](../patchnotes-release.md).
- **Direction:** If we later add tooling (e.g. optional generation or placeholder stripping), we can extend the Release line with a dated note instead of leaving ambiguous expectations.
