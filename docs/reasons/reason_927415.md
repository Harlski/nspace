# reason_927415 — Reposition ghost previews in THE-LARGER-SYSTEM

## Why this file exists

`docs/THE-LARGER-SYSTEM.md` was updated with a **normative principle** and a short **recorded decision** about **client-only translucent previews** when moving placeable obstacles in build mode, so future features do not ship reposition flows without visible hover feedback.

## What changed in THE-LARGER-SYSTEM

- New principle: **Reposition ghost previews** — any build-mode **move / reposition** of a placeable obstacle should show a **semi-transparent preview** at a valid hover target before commit; previews are **client-only** and do not imply server authority until the action is sent.
- New subsection under **Recorded decisions**: states how this aligns with existing implementations (gates, billboards, `makeBlockMesh` ghost path, placement preview) and sets expectation for **new** reposition surfaces.

## How this serves the larger system

- **Coherence:** One UX contract for “where will this land?” across object types, instead of gates being the only polished path.
- **Evolvability:** Implementers have an explicit checklist item when adding reposition for a new object kind (footprint + ghost or equivalent).
- **Direction:** Keeps authoring UX aligned with **server authority** — ghosts are visualization only, matching how we already treat overlays elsewhere in the doc.
