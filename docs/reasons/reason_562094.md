# reason_562094 — NIM rewards for gameplay goals (Free Play Field only) + server Goalies

## Why THE-LARGER-SYSTEM was updated

Until now, NIM left the treasury only for **claim-style** rewards (mineable blocks, feedback
tickets, manual/admin payouts) — never for an ordinary act of play. The World Cup feature
introduces the first **gameplay-earned** payout: scoring a goal in the shared **Free Play
Field** queues a small NIM payout (default 0.25 NIM) to the credited scorer. Paying real
money for a repeatable in-game action is a genuine economic boundary decision, so it is
recorded here rather than living only in feature code, per the repo handbook and
[worldcup/adr/0002](../../worldcup/adr/0002-nim-rewards-free-play-only.md).

The same change adds the first **server-controlled defender** (the Goalie), which standing
in every goal makes scoring (and therefore the payout) a skill rather than a tap.

## How it serves the larger system

- **Monetization coherence / guardrail:** Rewards are confined to the **Free Play Field**.
  Private 1v1 **Matches never pay**, because two colluding accounts could trade goals freely
  in isolation; the open field at least requires presence among strangers (the **Contested**
  guard, ≥2 distinct players). All thresholds are env-tunable so spend can be tuned without a
  code change: per-goal amount, per-wallet daily cap, global daily budget, and the minimum
  player count.
- **Idempotency reuses an existing seam:** A Paid Goal's `claimId`
  (`wc-goal-{wallet}-{utcDay}-{dailyIndex}`) is deterministic, so the existing
  `enqueueNimPayout` dedupe guarantees a goal is never paid twice on retries, and the
  per-wallet daily cap falls out of the same counter. Payouts flow through the **unchanged**
  payout queue (retries, dead-letter, history, admin surfaces), so this new source is
  monitored like any other.
- **Never blocks play:** Past the cap, over budget, uncredited, or off a Goalie deflection,
  the goal still counts for the leaderboard — only the payout stops. Attribution reuses the
  credited-last-kicker rule and excludes a sentinel Goalie address so keepers can't be farmed.
- **Authority + reversibility:** Goalies stay **server-authoritative** (a pure tracking model
  selected at runtime by `WORLDCUP_GOALIE_MODE`: `kicker` injects the keeper into the existing
  proximity-kick path; `blocker` is a circular collider inside `stepBall`). They broadcast
  lightweight positions alongside the existing throttled ball stream — consistent with the
  recorded dynamic-object decision ([reason_615240](reason_615240.md)). Everything remains
  flag-isolated under the seasonal `worldcup/` surface: `WORLDCUP_ENABLED=0` disables it and
  deleting the folder plus the `worldcup`-tagged hooks removes it entirely. The per-UTC-day
  reward counters live in a small, deletable JSON, reset forward-only on day rollover.
