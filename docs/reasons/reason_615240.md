# reason_615240 — Record server-simulated dynamic objects (soccer ball)

## Why THE-LARGER-SYSTEM was updated

The World Cup soccer feature introduces the first **server-simulated dynamic non-player
object** in Nimiq Space: a ball with continuous position + velocity that the
authoritative tick integrates (friction, wall/obstacle bounce) and broadcasts, plus
proximity-based "kick" detection derived from existing player positions. Until now the
world model only held **static** obstacles/floors/signage and **pathing NPCs** (which
reuse `PlayerState`). A moving physics object is a genuinely new world-model element, so
it warrants a recorded decision rather than living only in feature code and chat.

## How it serves the larger system

- **Coherence:** Records that dynamic objects are **server-authoritative**, follow the
  same intent-in / snapshot-out contract as players, and ship their own throttled
  delta-style broadcast (`ballState`) rather than overloading `state`.
- **Evolvability:** Captures the **feature-flag isolation + deprecation** expectation so
  a future seasonal/dynamic object follows the same removable pattern (`worldcup/`
  folders + a few greppable hook lines) instead of threading through core stores.
- **Direction:** If/when more dynamic objects appear, this is the anchor to decide
  whether to generalize into a shared dynamic-entity registry; today we deliberately keep
  it a bounded, deletable module so the experiment does not harden core contracts.
