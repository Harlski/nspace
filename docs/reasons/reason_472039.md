# Reason 472039 — Player-adjacent persistence in THE-LARGER-SYSTEM

**Date:** 2026-05-07  
**Doc touched:** [docs/THE-LARGER-SYSTEM.md](../THE-LARGER-SYSTEM.md)

## Why this update

Nimiq Space already stores several kinds of **account- or ops-related** data as **separate JSON files** on the game server. That pattern is acceptable while the surface area is small, but it does not scale cleanly: whole-file IO, weak concurrency semantics, and many ad hoc schemas make **future player state** (profiles, streaks, entitlements, moderation, etc.) harder to evolve without repeated rewrites.

The team wants a **shared mental model** before more player data features land: avoid painting the architecture into a corner where every feature adds another JSON document as the permanent source of truth.

## How it serves the larger system

A matching **Principles** bullet was added so the guidance shows up in the living principles list, not only under recorded decisions.

The new subsection under **Recorded decisions & forward constraints** does three things:

1. **Names the tradeoff** — JSON-on-disk is acknowledged as fine for early scale; the risks (IO pattern, concurrency, sprawl) are stated explicitly so they are not invisible debt.
2. **States a direction without mandating a vendor** — bounded persistence (with SQLite called out only as an **existing precedent** in the monorepo) and wallet-keyed rows, plus migrations, give implementers a default “good” without locking a single database engine forever.
3. **Encourages incremental migration** — dual-write / backfill / retire-file is explicitly allowed, which matches how a live service can evolve without a big-bang rewrite.

Future feature specs and code reviews can point here when deciding whether a new store belongs in a **shared** persistence layer or needs an explicit exception.
