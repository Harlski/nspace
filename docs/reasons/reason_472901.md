# Reason 472901 — THE-LARGER-SYSTEM: payment intent sidecar (completed scope)

> **Note:** Subsequent edits to the same subsection (JWT **`/api/payment/*`**, payer-scoped sidecar reads) are summarized in [reason_519027.md](reason_519027.md).

## Why `docs/THE-LARGER-SYSTEM.md` was updated

Incoming NIM monetization touches **authority**, **deploy topology**, and **security** (secrets, who may create intents, separation from the existing payout sender). That belongs in the larger-system doc so future work does not re-litigate what is already decided or half-shipped.

## What changed in THE-LARGER-SYSTEM

Added subsection **“Monetization: incoming NIM payment intents (sidecar — partial ship)”** that:

1. Lists what is **actually implemented** today (workspace, Docker/Compose, admin snapshot probes, tests).
2. States what is **deliberately not implemented** yet (no JWT proxy from `nspace` to the sidecar; no product entitlements).
3. Records **constraints** for the next phase (server-to-server only, JWT-bound payer, keep incoming verify separate from outgoing payouts).

## How this supports the larger system

- **Coherence:** One place distinguishes **infrastructure shipped** from **product not wired**, so [features-checklist.md](../features-checklist.md), [live-service-implementation.md](../live-service-implementation.md), and brainstorm files are not confused for each other.
- **Evolvability:** The next implementer can attach username / billboard / land / teleporter flows without redesigning the sidecar contract blindly.
- **Clear direction:** Preserves the architectural split (optional process, same monorepo) agreed when the sidecar was introduced.
