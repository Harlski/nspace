# reason_906712 — THE-LARGER-SYSTEM: authoring HUD / build dock

## Why this update

Ongoing work moved **build authoring chrome** toward a **single bottom dock**: category tabs (Terrain / Props / Buildings), **Objects / Room** scope when both per-room caps apply, compact **context** (modifiers + hue), and **docked GL previews** (placement + selection, including teleporter pillar in the selection slot). Tutorial-style **hint paragraphs** were removed from the build placement surface to match the existing principle **In-world UI copy stays idiomatic**.

That trajectory is **cross-cutting** for HUD density, the **Objects vs Room** mental model, and where future tools should land—so it belongs in [docs/THE-LARGER-SYSTEM.md](../THE-LARGER-SYSTEM.md), not only in the feature checklist.

## What changed in `THE-LARGER-SYSTEM.md`

1. **Principle (Objects vs Room)** — Clarified that the **client** keeps the **scope `<select>`** adjacent to **Terrain / Props / Buildings** (right side of the dock tab row), not a second “edit menu” beside **Build**.
2. **Recorded decision** — New subsection **Authoring HUD: bottom build dock (compact placement)** — describes today’s dock contract, the norm of one primary placement surface, and a forward constraint for new tools.

## How this serves the larger system

- **Coherence** — One row ties **what layer you edit** (Objects vs Room) to **what you place** (category tabs), so the HUD matches the mental model already stated for Objects vs Room.
- **Evolvability** — Future placement features have an explicit **hook** (dock contract); we avoid sprouting parallel build menus.
- **Direction** — Aligns **HUD copy density** with **In-world UI copy stays idiomatic**: controls stay terse; explanations live in docs and patch notes.
