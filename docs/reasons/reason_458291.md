# Reason 458291 — THE-LARGER-SYSTEM: production deploy backup

## What changed

[`docs/THE-LARGER-SYSTEM.md`](../THE-LARGER-SYSTEM.md) gained a **Recorded decisions** subsection **Production VPS deploy: stop, backup `data/`, then upgrade**, describing the current GitHub Actions flow: `docker compose stop` → tarball of host `data/` → git fast-forward → `docker compose build` / `up -d`, plus operator expectations (retention, restore sketch).

## Why

Shipping to `main` triggers an automated VPS refresh. Without an explicit, documented contract, operators can assume either that Docker “owns” persistence (it does not—**bind mounts** live on the host) or that deploys are safe to retry without a rollback artifact. Recording the **graceful stop → point-in-time archive → upgrade** sequence in the larger-system doc makes **durability and rollback** a first-class part of how we think about the live service, aligned with existing emphasis on **clear process** (e.g. `prepare-merge` for patch notes) and **honest operator-facing truth** (patch note tiers, deploy docs).

## How this serves the larger system

- **Coherence** — One place ties **code behavior** (SIGTERM flush in the server), **compose layout** (`./data` mount), and **CI** (workflow steps) so future changes to deploy or persistence stay consistent.
- **Evolvability** — When we add pre-update player notices, blue/green, or off-site backups, this subsection becomes the anchor to update rather than scattered README drift.
- **Direction** — Reinforces that **player-adjacent and world state on disk** deserve a predictable safety step before every image refresh.
