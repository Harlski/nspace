# reason_917384 — THE-LARGER-SYSTEM: build dock Room settings + scope reset

## Why this update

Follow-on **build dock** work tightened **Room** authoring UX and fixed a **stuck state** when authors closed build from **Room settings** and reopened: scope and tab memory stayed on **Room** even though walk mode should start fresh at **Objects**.

Room-level controls (**background hue**) now live in the **right context column** (same parameter surface as block height and hue), not a separate middle column—matching the **Objects vs Room** principle that room configuration is not mixed into the object tool strip.

**Mobile** authors get a **custom overlay** for **Objects / Room** instead of the native `<select>` sheet, with **opaque** menu rows for readability.

## What changed in `THE-LARGER-SYSTEM.md`

1. **Recorded decision (Authoring HUD: bottom build dock)** — Room **Floor / Room settings** tabs; **room background** in the **context** column; **mobile** scope overlay; **reset to Objects** when leaving or opening build (when both caps apply).
2. **Changelog** — Dated entry pointing here.

## How this serves the larger system

- **Coherence** — One **parameter box** for placement modifiers and room sky color; scope reset avoids ambiguous “half Room / half Objects” UI after dismiss.
- **Evolvability** — Documented reset contract so future dock tabs or scope controls do not reintroduce sticky state.
- **Direction** — Touch-friendly scope picker without breaking desktop native select; idiomatic compact chrome.
