# reason_248173 — Per-room straight-line (non-pathfinding) movement on the soccer pitch

## Why THE-LARGER-SYSTEM was updated

Human movement in Nimiq Space has always been click-to-tile with **cardinal grid
pathfinding** (`pathfindTerrain`): destinations snap to integer tile centers and paths are
axis-aligned segments. That made the World Cup ball nearly unkickable at angles — because
the ball's kick heading is derived from `player.vx/vz`, which was almost always pure ±X or
±Z. The soccer pitch now uses a **straight-line, non-pathfinding** move to the exact
clicked float point (clamped to the open rectangle). This is the **first per-room movement
variant**, a real change to a long-standing core behavior, so it warrants a recorded
decision rather than living only in feature code.

## How it serves the larger system

- **Coherence:** The variant stays **server-authoritative** and reuses the existing
  per-tick follower (`advanceAlongPathHuman`); only the destination source changes (no
  `snapToTile`, no BFS, drift-snap skipped on the field). The client merely sends the raw
  float floor-hit and draws a straight preview; doors still use the snapped standing-tile
  check, so the rest of the navigation contract is untouched.
- **Guardrail:** Records that free movement is justified **only** by an obstacle-free room
  where straight lines can never clip geometry. It explicitly should **not** be extended to
  rooms with placed obstacles without a proper collision model.
- **Evolvability / reversibility:** Kept flag-isolated under the seasonal `worldcup/`
  surface — turning the flag off reverts the field to standard grid movement, and deleting
  the feature removes the branch entirely. Client-only pitch visuals (ball texture, grass +
  markings, nets, stands) are raycast-disabled decoration that never affect picking or
  collision, consistent with the existing decorative-overlay principle.
