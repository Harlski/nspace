# Reason 731654 — THE-LARGER-SYSTEM: route parity for new pages on the split SPA host

## What changed

[`docs/THE-LARGER-SYSTEM.md`](../THE-LARGER-SYSTEM.md) gained:

- A **principle** — *New routed surfaces ship with split-host routing*: any new server-rendered HTML route or new clean-path client route must add its rewrite to **both** [`vercel.json`](../../vercel.json) and [`client/vercel.json`](../../client/vercel.json) in the same change, because the static SPA host has **no catch-all fallback**.
- A **Recorded decisions** subsection — *Split SPA hosting: route parity for new pages* — documenting how the two `vercel.json` files enumerate explicit rewrites (server HTML pages → `https://api.nimiq.space/<path>`; clean-path client routes → built HTML; `/api/*` and `/nim-chart-api/*` proxy JSON; static MPA artifacts served directly), plus the forward constraint to keep [live-service-implementation.md](../live-service-implementation.md) in sync and verify both files parse.

## Why

The `/admin/rooms` page worked on the all-in-one dev/VPS server but would have **404'd on Vercel** because no rewrite existed for it — every server-rendered page is listed individually and there is no SPA catch-all. This is an easy, recurring trap: a feature looks complete locally yet is unreachable in production hosting. Encoding "add the rewrite in the same change" as a durable principle (not a one-off fix) prevents the next new page from regressing the same way.

## How this serves the larger system

- **Coherence** — Ties three things together (server route registration in `server/src/index.ts`, the two `vercel.json` rewrite tables, and the route list in `live-service-implementation.md`) so deploy topology stays consistent as pages are added.
- **Evolvability** — Gives a single anchor to update if the hosting split, the API host, or the catch-all policy changes (e.g. if a future SPA fallback is introduced, this is where to relax the rule).
- **Direction** — Reinforces that "done" for a user-facing route includes **production reachability on the split host**, matching the project's existing emphasis on honest operator-facing truth and deploy process.
