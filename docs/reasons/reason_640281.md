# Reason 640281 — Objects vs Room in authoring UI

## Why `THE-LARGER-SYSTEM.md` was updated

Guest join spawn and similar controls were grouped with **placeable objects** in the build tool UI, which blurred two different concepts: **things in the world** versus **settings of the room itself**. We aligned the product with a clear split and recorded it so future features (permissions, docs, new room toggles) stay consistent.

## How this supports the larger system

- **Coherence** — Players and contributors see one stable rule: room ambience, spawn policy, and room-scoped layout affordances live under **Room**; blocks and interactive props live under **Objects**.
- **Evolvability** — New room-level options can extend the Room surface without inflating the object palette or overloading block-placement code paths.
- **Direction** — Server authority for `joinSpawn` and room metadata already treats these as room fields; the UI principle matches that model instead of mimicking “another block tool.”
