# reason_392847 — Per-tile floor color in the build dock

## Why `THE-LARGER-SYSTEM` was updated

Nimiq Space already treated **room-level** presentation (sky hue, guest spawn, extra floor topology) as **Room** scope and **placeable content** as **Objects**. Authors can pick any **`colorRgb`** for blocks via the shared hue ring, but floor cells were still mostly **fixed palette tones** until this release.

## How this supports the larger system

- **Objects vs Room:** Floor **paint** is room topology/presentation, not a placed obstacle—so it lives under **Room → Floor** with the same **hue ring + hex popover** contract as blocks, instead of a one-off color picker or a fake “floor block” tool.
- **Scale simple → rich:** Per-tile floor state is **`colorRgb` on existing `placeExtraFloor` / persistence paths**, with **`baseFloorColors`** for core grid and legacy string merge—no new whole-file JSON silo; authority stays on the server.
- **Tiles forward constraint:** The “per-cell customization” note now reflects that **solid floor color per cell** is shipped; future mesh/decal work can split **tile data** from **rendering** without undoing this contract.

## What changed in the doc

- **Authoring HUD** subsection — floor hue ring, recolor-under-blocks, left/right click floor edit.
- **Tiles: per-cell customization** — “today” updated from color-only aspiration to implemented **`colorRgb`** on core and extra walkable tiles.
