# 03 — 1v1 scorer stuck after own goal until opponent scores (Pay suspected)

## Report

In the **World Cup 1v1 Match Pitch**, after **scoring a goal**, the scoring player becomes **unable to move** until the **other player scores**. Movement resumes when the opponent scores. Reporter unsure if desktop is affected; may be **Nimiq Pay only**.

## Triage notes

> *This was generated by AI during triage.*

**Redundancy check:** Post-goal kickoff freeze is intentional — both players should be immobile for `goalResetMs`, then resume. This bug is **extended or one-sided** freeze beyond that window.

**Prior rejection:** None.

**Codebase summary:**

**Server**

- On match goal (non-terminal): `worldcupKickoffReset` snaps players, sets `m.kickoffUntilMs = now + goalResetMs`, rejects `moveTo` while `kickoffUntilMs > now`.
- Match tick loop pauses clock during kickoff freeze; clears `kickoffUntilMs` when expired.

**Client**

- On `matchGoal` with `kickoffMs > 0`: `setWorldcupMoveLocked(true)` then `setTimeout(..., kickoffMs)` → `setWorldcupMoveLocked(false)`.
- Client lock blocks local joystick/tap emit; server also rejects moves during kickoff window.
- `matchEnded` clears lock unconditionally.

**Hypotheses**

1. **Timer throttling** in Pay WebView: `setTimeout` for unlock delayed or dropped; scorer stays client-locked while server already accepts moves (or vice versa).
2. **Client/server desync**: client unlocks but server still in kickoff (unlikely until opponent scores).
3. **Scorer-only path**: something leaves `worldcupMoveLocked` true for scorer but not opponent until next `matchGoal` resets timers for both.
4. Overlapping goal events clobber unlock timers.

**Verification:** Not reproduced in triage. Needs Pay device repro; confirm desktop Hub browser too.

**Recommendation:** `bug` + `ready-for-agent` (with manual Pay verification in acceptance criteria).

## Agent Brief

**Category:** bug  
**Summary:** After a 1v1 match goal, both players must regain movement when the kickoff countdown ends — not only when the next goal is scored.

**Current behavior:** The player who scored remains stuck (cannot move) after the post-goal kickoff period. They can move again only after the opponent scores another goal. Opponent may or may not be affected (confirm during fix).

**Desired behavior:** After each non-terminal match goal, both participants are frozen for the configured kickoff duration, then **both** can move regardless of who scored. Behavior matches on desktop Hub and Nimiq Pay.

**Key interfaces:**

- `matchGoal` WebSocket message — carries `kickoffMs`; both clients must unlock in sync with server
- `setWorldcupMoveLocked` — client-side movement gate during kickoff
- Match runtime `kickoffUntilMs` — server-side move rejection window
- Match HUD kickoff countdown — uses `performance.now()` + interval; consider aligning unlock with same clock source

**Acceptance criteria:**

- [ ] Score a goal in 1v1; after kickoff countdown, **scorer** can move without waiting for opponent goal
- [ ] Opponent can also move after same kickoff window
- [ ] Multiple consecutive goals do not leave either player permanently locked
- [ ] Match-ending goal still transitions to result flow (no spurious unlock mid-teardown)
- [ ] Verified on Nimiq Pay mini-app; note whether desktop reproduces
- [ ] `npm run build` and relevant server worldcup match tests pass

**Out of scope:**

- Changing kickoff duration defaults
- Free Play Field goals (different message path)

## Comments

### 2026-06-21 — implemented

**Root cause:** Client unlock used a bare `setTimeout(kickoffMs)` while the Match HUD countdown used `performance.now()` + `setInterval`. Pay WebView can throttle or drop one-shot timers; the HUD interval often still runs, leaving `worldcupMoveLocked` true after kickoff while the server had already cleared `kickoffUntilMs`. Opponent's next `matchGoal` accidentally fixed it by scheduling another unlock attempt.

**Fix:**
- Unlock from the HUD kickoff countdown end callback (same timer as "Kickoff in N…").
- Server broadcasts `kickoffRemainingMs` on `matchState` (including during kickoff freeze ticks) — client unlocks when it hits 0 as authoritative fallback.
- Removed `setTimeout` unlock path.

**Verify on Pay:** score in 1v1; after kickoff countdown both players move without waiting for another goal.
