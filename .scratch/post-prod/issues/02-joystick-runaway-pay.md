# 02 — Touch joystick runaway after finger release (Nimiq Pay)

## Report

On **Nimiq Pay** (touch / coarse pointer), releasing the **Touch Joystick** finger does not stop the player — they keep running in the last direction indefinitely. Expected: movement stops the instant the finger lifts.

## Triage notes

> *This was generated by AI during triage.*

**Redundancy check:** Release-to-stop is **already implemented** — not a missing feature. `Game.endWorldcupStick()` clears the emit interval and calls `worldcupJoystickStop()`, which sends an un-rate-limited `stop` move intent via `sendStopMove`. Do not re-implement from scratch; fix the broken path.

**Prior rejection:** None.

**Codebase summary:**

- Touch Joystick visual: floating stick anchored at thumb-down (`WorldcupJoystick`).
- Gesture + emit loop lives in `Game.ts`: `beginWorldcupStick` starts a `setInterval` re-emitting heading; `endWorldcupStick` tears down on release.
- Release handling: `onPointerUp` / `pointercancel` on the **canvas** calls `endWorldcupStick()` when `worldcupStick.pointerId` matches.
- Separate `onWindowTouchPointerEnd` on `window` (capture) only calls `releaseTouchPointerId` — **does not** end the joystick session.
- In Pay WebView, finger lift may fire on `window` or over HUD chrome without the canvas `pointerup` reaching the stick handler (especially if capture is lost or the finger slides off the canvas).

**Verification:** Not reproduced in triage (requires Pay device). Strong code-path hypothesis: window-level touch end must also call `endWorldcupStick()` for the active stick pointer.

**Recommendation:** `bug` + `ready-for-agent`.

## Agent Brief

**Category:** bug  
**Summary:** Releasing the Touch Joystick must always stop the player immediately on touch hosts (especially Nimiq Pay).

**Current behavior:** After steering with the floating Touch Joystick, lifting the finger sometimes leaves the player moving continuously in the last direction. The client keeps re-emitting move intents from the stick interval until another input interrupts.

**Desired behavior:** Any finger release, cancel, or gesture termination that ends the stick pointer stops movement immediately — same as a deliberate `stop` intent today when `endWorldcupStick` runs. Works on canvas release, release over HUD, `pointercancel`, tab hide, and Pay WebView edge cases.

**Key interfaces:**

- `Game.endWorldcupStick()` — single teardown + stop path; all release routes should converge here
- `Game.onWindowTouchPointerEnd` — window capture listener for touch pointer end/cancel
- `sendStopMove` / move `stop: true` — server clears path without rate-limit
- `WorldcupJoystick` enable gate — touch host + pitch + not spectating

**Acceptance criteria:**

- [ ] Releasing the stick finger on the canvas stops movement immediately
- [ ] Releasing while the finger is over HUD / outside canvas still stops movement
- [ ] `pointercancel` stops movement
- [ ] No stuck interval after `visibilitychange` / `pagehide` (existing flush should remain)
- [ ] Tap-to-move still works alongside the stick
- [ ] Manual verify on Nimiq Pay (or `?payEmulate` + coarse pointer + DevTools touch)
- [ ] `npm run build` passes

**Out of scope:**

- Changing joystick visual design or placement
- Desktop mouse movement

## Comments

### 2026-06-21 — follow-up fix (root cause)

**Root cause:** `setWorldcupStopMoveHandler(() => sendStopMove(socket))` referenced undefined `socket` (typo for `ws`). Joystick release cleared the client interval but **never sent** `{ type: "moveTo", stop: true }` — server kept the path and client extrapolated velocity indefinitely.

**Fix:** Use `ws` closure (`const s = ws; if (s) sendStopMove(s)`). Zero `selfServerVx`/`selfServerVz` in `worldcupJoystickStop()` for immediate local halt before the server ack.

Window-level `endWorldcupStick` (prior commit) remains useful when canvas misses pointerup.
